## 1. Philosophie

### Ce qui change entre 2016 et 2026

Le systeme legacy PrediabAid a ete concu dans un monde sans LLM. Toute la personnalisation passait par des **regles deterministes** : 430 regles Drools, 5 formules, 20 combinaisons age × IMC, 8 phases a durees fixes. C'etait la seule approche possible.

En 2026, les LLM permettent de :
- Comprendre du texte libre (plus besoin de 40 questionnaires a choix multiples)
- Analyser des images (photo d'un repas → tracking automatique)
- Generer du contenu personalise (plus besoin de 430 regles de selection de videos)
- Converser naturellement (plus besoin de formulaires rigides)
- Raisonner sur un contexte riche (200K tokens = tout l'historique d'un patient dans un prompt)

### Deux couches, deux roles

```
┌──────────────────────────────────────────────┐
│           SOCLE DETERMINISTE                  │
│                                               │
│  Seuils medicaux, exclusions, alertes,       │
│  contraintes nutritionnelles, solveur LP      │
│                                               │
│  → Auditable, testable, non negociable        │
│  → JSON as code, pas de LLM                   │
└───────────────────────┬──────────────────────┘
                        │ garde-fous
┌───────────────────────▼──────────────────────┐
│           COUCHE IA                           │
│                                               │
│  NE REFLECHIT PAS. REDIGE.                   │
│  Recoit des donnees calculees, les met en    │
│  forme en langage naturel.                    │
│                                               │
│  → LLM avec donnees pre-machees + validation  │
│  → Jamais de decision medicale                │
└──────────────────────────────────────────────┘
```

### Regle fondamentale : le LLM ne reflechit pas, il redige

Le LLM n'est pas le cerveau du systeme. C'est la voix. Le cerveau c'est le moteur de regles, le solveur LP, les calculs statistiques. Le LLM prend un resultat calcule et le formule en langage naturel.

| Mauvais (le LLM decide) | Bon (le LLM redige) |
|---|---|
| "Je pense que votre glycemie est un peu elevee" | "Votre glycemie est a 130 mg/dL. Le seuil d'alerte est 126." (calcule) |
| "Je vous recommande du saumon ce soir" | "Votre menu du soir (genere par le solveur) : saumon + quinoa (520 kcal)" |
| "Vous avez perdu environ 1 kg" | "Cette semaine : -0.7 kg (-0.83%)" (calcule par pandas) |
| "Je pense que vous etes prete pour la stabilisation" | "Le systeme a evalue : poids stable 3 semaines + compliance 82% → STABILISATION" |

## 2. Pattern "Template enrichi"

### Le probleme

Deux extremes, les deux mauvais :
- **Template pur** (legacy) : "Bonjour {prenom}, jour {jour} de votre phase {phase}" → robotique, generique, zero empathie
- **LLM libre** (ChatGPT) : repond ce qu'il veut → hallucinations, incoherences, non reproductible

### La solution : template enrichi

4 etapes, chaque etape a un role strict :

```
┌──────────────────────────────────────────────────────┐
│  ETAPE 1 — CALCUL (Python, 100% deterministe)        │
│                                                       │
│  glycemie_mean = 102 (pandas)                        │
│  glycemie_alert = False (regle JSON : 102 < 126)     │
│  weight_delta = -0.7 (calcul)                        │
│  compliance = 78% (comptage)                         │
│  tir = 72% (pandas)                                  │
│  sport_done = 3, sport_target = 4 (comptage)         │
│  phase = PROGRESSION, day = 18, total = 28           │
│  menu_today = {recipe_ids from solver}               │
│  triggers_detected = ["fatigue_soir"] (stats)        │
│                                                       │
│  → Aucun LLM. Tout est calculable et verifiable.     │
└───────────────────────┬──────────────────────────────┘
                        │
                        ▼
┌──────────────────────────────────────────────────────┐
│  ETAPE 2 — STRUCTURE (Python, deterministe)           │
│                                                       │
│  Selectionne le template + injecte les donnees :     │
│                                                       │
│  {                                                    │
│    "template": "weekly_bilan",                       │
│    "data": {                                         │
│      "patient_name": "Marie",                        │
│      "weight_delta_kg": -0.7,                        │
│      "weight_delta_pct": -0.83,                      │
│      "compliance_pct": 78,                           │
│      "tir_pct": 72,                                  │
│      "sport_done": 3, "sport_target": 4,             │
│      "phase": "PROGRESSION", "phase_day": 18,        │
│      "positive_point": "weight_loss_on_track",       │
│      "improve_point": "sport_frequency",             │
│      "behavioral_trigger": "evening_fatigue"         │
│    },                                                │
│    "instructions": [                                 │
│      "Reformule ce bilan en langage naturel",        │
│      "Ton bienveillant et factuel",                  │
│      "NE CHANGE AUCUN CHIFFRE",                      │
│      "NE RAJOUTE AUCUNE info medicale",              │
│      "Utilise UNIQUEMENT les donnees fournies",      │
│      "Max 150 mots"                                  │
│    ]                                                 │
│  }                                                    │
│                                                       │
│  → Les FAITS sont fixes. Le LLM recoit un brief.     │
└───────────────────────┬──────────────────────────────┘
                        │
                        ▼
┌──────────────────────────────────────────────────────┐
│  ETAPE 3 — REDACTION (LLM)                            │
│                                                       │
│  Le LLM recoit le brief structure et redige :        │
│                                                       │
│  "Marie, bonne semaine ! Vous avez perdu 700g et     │
│   suivi 78% de vos menus. Votre TIR est a 72%,      │
│   juste au-dessus de l'objectif de 70%.              │
│                                                       │
│   Le midi, c'est tres bien suivi. Le soir reste      │
│   plus difficile — la fatigue de fin de journee      │
│   est votre principal declencheur.                   │
│                                                       │
│   Cette semaine : on vise la 4eme seance de sport.   │
│   Meme 15 minutes de marche apres le diner compte."  │
│                                                       │
│  → Le LLM a REFORMULE, pas INVENTE.                  │
└───────────────────────┬──────────────────────────────┘
                        │
                        ▼
┌──────────────────────────────────────────────────────┐
│  ETAPE 4 — VALIDATION (Python, 100% deterministe)     │
│                                                       │
│  Verifications automatiques :                        │
│  ✓ Les chiffres dans le texte matchent l'input      │
│    (-0.7 kg, 78%, 72%, 3/4)                          │
│  ✓ Aucun nom de medicament mentionne                 │
│  ✓ Aucun diagnostic ("vous avez", "vous etes")       │
│  ✓ Pas de conseil hors scope                         │
│  ✓ Longueur < 150 mots                              │
│                                                       │
│  Si echec → fallback sur template statique :         │
│  "Bilan semaine : poids -0.7 kg, compliance 78%,    │
│   TIR 72%, sport 3/4. Continuez !"                   │
│                                                       │
│  → Garantie : jamais de contenu non valide envoye.   │
└──────────────────────────────────────────────────────┘
```

### Validation : implementation

```python
def validate_llm_output(
    response: str,
    input_data: dict,
    template_type: str
) -> ValidationResult:
    """Verifie que la sortie LLM est fidele aux donnees d'entree."""

    checks = []

    # 1. Verification des chiffres
    # Extraire tous les nombres du texte genere
    # et verifier qu'ils existent dans input_data
    numbers_in_response = extract_numbers(response)
    numbers_in_input = extract_all_values(input_data)

    for num in numbers_in_response:
        if not matches_any(num, numbers_in_input, tolerance=0.01):
            checks.append(Check(
                passed=False,
                reason=f"Chiffre {num} absent des donnees d'entree"
            ))

    # 2. Pas de medicaments
    MEDICATION_PATTERNS = [
        r"metformin", r"insuline", r"glibenclamide",
        r"medicament", r"comprime", r"traitement",
    ]
    for pattern in MEDICATION_PATTERNS:
        if re.search(pattern, response, re.IGNORECASE):
            checks.append(Check(
                passed=False,
                reason=f"Mention de medicament detectee: {pattern}"
            ))

    # 3. Pas de diagnostic
    DIAGNOSIS_PATTERNS = [
        r"vous (etes|avez) diabetique",
        r"vous souffrez de",
        r"je diagnostique",
        r"c'est (normal|pas grave|inquietant)",
    ]
    for pattern in DIAGNOSIS_PATTERNS:
        if re.search(pattern, response, re.IGNORECASE):
            checks.append(Check(
                passed=False,
                reason=f"Diagnostic potentiel detecte: {pattern}"
            ))

    # 4. Longueur
    word_count = len(response.split())
    max_words = TEMPLATE_MAX_WORDS.get(template_type, 200)
    if word_count > max_words:
        checks.append(Check(
            passed=False,
            reason=f"Trop long: {word_count} mots (max {max_words})"
        ))

    return ValidationResult(
        passed=all(c.passed for c in checks),
        checks=checks
    )
```

### Fallback

Si la validation echoue, le systeme utilise un **template statique** — moins elegant mais 100% sur :

```python
FALLBACK_TEMPLATES = {
    "weekly_bilan": (
        "Bilan semaine : poids {weight_delta_kg:+.1f} kg, "
        "compliance {compliance_pct}%, "
        "TIR {tir_pct}%, "
        "sport {sport_done}/{sport_target} seances."
    ),
    "daily_message": (
        "Bonjour {patient_name}. "
        "Phase {phase} — jour {phase_day}. "
        "Menu du jour disponible."
    ),
    "glycemia_alert": (
        "Votre glycemie est a {value} mg/dL. "
        "Le seuil d'alerte est {threshold} mg/dL. "
        "Consultez votre medecin."
    ),
}
```

Le patient recoit toujours un message. Soit redige par le LLM (quand valide), soit un template statique (quand le LLM echoue). Jamais de silence, jamais de contenu non verifie.

### Ou le pattern s'applique

| Fonctionnalite | Etape 1 (calcul) | Etape 2 (structure) | Etape 3 (LLM) | Etape 4 (validation) |
|---|---|---|---|---|
| **Bilan hebdomadaire** | pandas : moyennes, deltas, compliance | Template `weekly_bilan` + data | Reformulation naturelle | Verification chiffres + scope |
| **Bilan mensuel** | pandas : trends, comparaisons | Template `monthly_bilan` + data | Narratif plus detaille | Verification chiffres + scope |
| **Bilan CGM** | pandas : TIR, TAR, TBR, patterns | Template `cgm_report` + data | Explication patient + resume praticien | Verification seuils + scope |
| **Message quotidien** | Regles JSON → DayData | Template `daily_message` + data | Personnalisation du ton | Verification scope |
| **Explication menu** | Solveur LP → recette + nutrition | Template `menu_explanation` + data | Texte explicatif | Verification calories + scope |
| **Reponse coach** | Recherche contexte patient | Context + tools disponibles | Conversation naturelle | Safety check + scope |
| **Photo repas** | — | Photo + menu prevu | Vision : identification | Verification coherence |

### Le cas special : conversation libre avec le coach

La conversation est le seul cas ou le LLM a plus de liberte. Mais meme la, il est contraint :

```python
# Le contexte injecte dans CHAQUE message du coach
coach_context = {
    # Donnees calculees (le LLM ne les invente pas)
    "patient": patient_profile,           # calcule
    "phase": current_phase,               # calcule
    "today_menu": today_menu,             # solveur LP
    "glycemia_status": glycemia_check,    # regles JSON
    "compliance": weekly_compliance,      # comptage
    "recent_triggers": behavioral_tags,   # extraction structuree

    # Ce que le LLM peut faire
    "available_tools": [
        "get_menu", "substitute_recipe",
        "log_sport", "set_reminder",
        "get_bilan", "get_recipe_details"
    ],

    # Ce que le LLM ne peut PAS faire
    "constraints": [
        "Ne jamais inventer de chiffre",
        "Ne jamais faire de diagnostic",
        "Ne jamais recommander de medicament",
        "Si question hors scope → 'Parlez-en a votre medecin'",
        "Si incertain → 'Je ne suis pas sur, je prefere ne pas
         vous donner une information incorrecte'"
    ]
}
```

Quand le patient demande "ma glycemie est a 130, c'est normal ?", le coach :
1. Regarde `glycemia_status` (calcule par le backend) → ALERTE (130 >= 126)
2. Ne dit PAS "c'est normal" (il sait que le backend dit ALERTE)
3. Dit : "Votre glycemie est a 130 mg/dL, au-dessus du seuil de 126. Ce n'est pas un diagnostic, mais je vous recommande d'en parler a votre medecin."

Le LLM n'a pas decide que 130 > 126. Le backend l'a calcule et injecte dans le contexte.

### Ratio LLM / deterministe par fonctionnalite

| Fonctionnalite | % Deterministe | % LLM | Role du LLM |
|---|---|---|---|
| Scoring PT → Groupe → Formule | 100% | 0% | — |
| Transition de phase | 100% | 0% | — |
| Generation menu (solveur LP) | 100% | 0% | — |
| Budget calorique | 100% | 0% | — |
| Seuils glycemie / alertes | 100% | 0% | — |
| Calcul TIR, TAR, TBR | 100% | 0% | — |
| Compliance (%) | 100% | 0% | — |
| Detection patterns comportementaux | 90% | 10% | Extraction de tags des conversations |
| Bilan hebdomadaire | 80% | 20% | Reformulation des donnees calculees |
| Explication d'un menu | 80% | 20% | Mise en forme des donnees nutritionnelles |
| Message quotidien | 70% | 30% | Personnalisation du ton et du timing |
| Conversation avec le coach | 40% | 60% | Dialogue, empathie, mais contraint par le contexte |
| Photo repas | 20% | 80% | Vision (pas de substitut deterministe) |
| Onboarding conversationnel | 30% | 70% | Conversation, extraction structuree |
| **Moyenne ponderee** | **~80%** | **~20%** | |

**Le produit est 80% deterministe, 20% LLM.** Le LLM est la couche de vernis qui rend le produit humain. Le moteur en dessous est 100% calculable, testable et auditable.

| Legacy | Probleme | Remplacement 2026 |
|--------|----------|-------------------|
| 5 formules × 4 segments age/IMC = 20 combinaisons | Arbitraire, pas personnalise | Programme adaptatif (durees selon progression reelle) |
| 8 phases a durees fixes + behaviors (JUMP, LOOP) | Rigide, patches sur patches | 4 phases a durees dynamiques |
| 430 regles de contenu (quelle video quand) | Du content management code en dur | Coach IA qui selectionne le contenu pertinent |
| 40+ questionnaires comme interface quotidienne | Friction, abandon | Photo repas + conversation |
| Scoring PT (somme de points → G1/G2/G3) | Simpliste, 25 questions fastidieuses | Onboarding conversationnel + scoring en arriere-plan |
| Messages pre-ecrits generiques | Pas personnalises | Bilans narratifs generes, coaching contextualise |
| 29 types de questions (SliderRadioTime...) | Complexite technique inutile | Conversation naturelle, le LLM extrait les donnees |

## 4. Modules IA

### 3.1 Photo du repas → tracking automatique

**Probleme resolu** : remplir 5 questionnaires repas par jour est la premiere cause d'abandon.

**Fonctionnement** :

```
📸 Patient prend une photo de son assiette
        │
        ▼
  API : POST /api/v1/meals/photo
  Body : { image: base64, meal_type: "lunch", date: "2026-03-19" }
        │
        ▼
  FastAPI → Claude Vision (Haiku)
  System prompt :
    "Tu es un nutritionniste. Analyse cette photo de repas.
     Identifie chaque aliment, estime la portion en grammes,
     calcule les calories et macronutriments.
     Compare avec le menu prevu : {menu_du_jour}.
     Reponds en JSON structure."
        │
        ▼
  Reponse structuree :
  {
    "items": [
      { "name": "saumon grille", "portion_g": 150, "kcal": 280, "proteins_g": 35 },
      { "name": "quinoa", "portion_g": 120, "kcal": 140, "carbs_g": 25 },
      { "name": "brocolis vapeur", "portion_g": 100, "kcal": 35, "fiber_g": 5 }
    ],
    "total_kcal": 455,
    "menu_match": {
      "planned_recipe": "salmon.quinoa.broccoli",
      "followed": true,
      "deviation_pct": 2
    }
  }
        │
        ▼
  → Compliance trackee automatiquement
  → Message coach : "Bon choix ! Pile dans votre budget."
  → Si ecart : "Je vois des pates blanches au lieu du quinoa.
     Pas grave, mais demain on revient au menu ?"
```

**Fallback** : si la photo est floue ou ambigue, le coach demande des precisions en conversation. Si le patient prefere, il peut toujours saisir manuellement.

**Modele** : Claude Haiku (vision). Cout : ~$0.01/photo × 3 photos/jour × 30 jours = **~$0.90/user/mois**.

### 3.2 Coach conversationnel

**Probleme resolu** : le patient est seul face a son programme, les messages generiques ne motivent pas.

**Architecture** :

```
┌──────────────────────────────────────────────┐
│              COACH IA                         │
│                                               │
│  ┌─────────────┐     ┌─────────────────────┐│
│  │ Context      │     │ Claude Sonnet       ││
│  │ Builder      │────►│                     ││
│  │              │     │ System prompt :      ││
│  │ • Profil     │     │ "Tu es un coach     ││
│  │ • Phase      │     │  sante bienveillant ││
│  │ • Mesures    │     │  et precis.         ││
│  │ • Menus      │     │  Voici le contexte  ││
│  │ • Compliance │     │  de ce patient..."  ││
│  │ • Humeur     │     │                     ││
│  │ • Historique │     │ + Tools disponibles  ││
│  └─────────────┘     └──────────┬──────────┘│
│                                  │           │
│                       ┌──────────▼──────────┐│
│                       │ Tools               ││
│                       │ • get_menu()        ││
│                       │ • substitute_recipe()│
│                       │ • get_glycemia()    ││
│                       │ • set_reminder()    ││
│                       │ • get_bilan()       ││
│                       │ • log_sport()       ││
│                       └──────────┬──────────┘│
│                                  │           │
│                       ┌──────────▼──────────┐│
│                       │ Safety Layer        ││
│                       │ (voir section 6)    ││
│                       └─────────────────────┘│
└──────────────────────────────────────────────┘
```

**Contexte injecte** (dans le system prompt, a chaque message) :

```
Patient : Marie, 52 ans, femme
Phase : PROGRESSION, jour 18/28 (adaptatif)
Poids : 82.8 kg (objectif sante : 78 kg, delta : -4.8 kg)
Glycemie (si CGM) : TIR 72%, moyenne 102, tendance -3%
Menu du jour : [petit-dej: porridge myrtilles, dej: poulet basquaise, diner: soupe legumes + pain complet]
Compliance semaine : menus 78%, sport 3/4 seances
Humeur recente : moyenne 6.2/10
Derniere interaction : hier, a demande une substitution pour le diner
Alertes actives : aucune
```

**Cas d'usage** :

| Situation | Comportement du coach |
|-----------|----------------------|
| Patient envoie un message libre | Repond en contexte, utilise les tools si besoin |
| Patient demande une substitution | Appelle `substitute_recipe()`, propose des alternatives |
| Patient dit qu'il est fatigue/demotive | Adapte le ton, propose un objectif reduit, encourage |
| Patient demande un conseil medical | Refuse poliment, renvoie au medecin |
| Patient annonce un ecart (diner dehors) | Pas de jugement, conseils pratiques, ajuste le menu |
| Patient ne repond plus depuis 3 jours | Message proactif via workflow Inngest |

**Modele** : Claude Sonnet pour les conversations, Haiku pour les messages proactifs quotidiens.

**Cout** : ~5 messages/jour × ~1000 tokens × 30 jours = **~$0.10/user/mois** (Haiku) a **~$0.50/user/mois** (mix Sonnet/Haiku).

### 3.3 Bilans narratifs

**Probleme resolu** : un tableau de chiffres ne dit rien a un patient. Un praticien n'a pas le temps de rediger un bilan pour chaque patient.

**Fonctionnement** :

```
Donnees collectees (7 derniers jours)
  │
  ▼
Calcul des metriques (Python, deterministe)
  • poids : 83.5 → 82.8 (-0.7 kg, -0.83%)
  • glycemie (si CGM) : TIR 72%, mean 102, trend -3%
  • compliance menus : 27/35 repas suivis (78%)
  • sport : 3 seances, 105 min (cible : 4 seances)
  • humeur : moyenne 6.2/10
  • questionnaires completes : 6/7 jours
  │
  ▼
Template de prompt :
  "Genere un bilan hebdomadaire pour ce patient.
   Donnees : {metriques}.
   Profil : {contexte patient}.
   Ton : bienveillant, precis, actionnable.
   Structure : 1 paragraphe resume, 1 point positif,
   1 point a ameliorer, 1 objectif pour la semaine."
  │
  ▼
Claude Haiku
  │
  ▼
"Marie, belle semaine ! Vous avez perdu 700g et votre
 glycemie moyenne a baisse de 3% — c'est exactement
 la tendance qu'on cherche.

 Votre point fort : les menus du midi sont bien suivis
 (90% de compliance le midi vs 65% le soir). Le soir
 est votre point faible, probablement la fatigue de
 fin de journee.

 Cette semaine, un seul objectif : ajoutez une 4eme
 seance de sport. Meme 15 minutes de marche apres
 le diner. Ca aidera aussi votre glycemie du soir."
```

**Types de bilans** :

| Type | Frequence | Genere par | Validation praticien |
|------|-----------|------------|---------------------|
| Hebdomadaire | Chaque lundi | Haiku (template) | Non |
| Mensuel | Chaque 1er du mois | Sonnet (narratif complet) | Optionnel |
| Fin de phase | A chaque transition | Sonnet (comparaison avant/apres) | Oui |
| Bilan CGM (fenetre 14j) | Apres chaque fenetre CGM | Sonnet (analyse patterns) | Oui |

### 3.7 Personnalisation menus (ML — phase 2)

**Probleme resolu** : le solveur LP optimise la nutrition, pas les gouts.

**Architecture** :

```
Score final recette = α × Score_LP + β × Score_ML + γ × Score_CGM

  α = 0.6  Score nutritionnel (solveur LP, deterministe)
  β = 0.3  Score preference (ML, apprentissage)
  γ = 0.1  Score glycemique (CGM correlation, si disponible)
```

**Score ML** (content-based filtering au debut) :

```
Signaux collectes :
  • Repas suivi (photo match menu)      → positif fort
  • Repas substitue                      → negatif fort
  • Recette ignoree (pas de photo)       → negatif faible
  • Note explicite ("j'ai pas aime")     → negatif fort
  • Ingredients en commun avec recettes aimees → positif moyen

Modele :
  • Phase 1 (< 1000 users) : embeddings recettes (ingredients,
    categorie, type repas, kcal) + similarite cosinus
  • Phase 2 (> 1000 users) : collaborative filtering
    (users similaires aiment des recettes similaires)
```

**Score CGM** (si fenetre CGM disponible) :

```
Pour chaque recette deja consommee pendant une fenetre CGM :
  • Mesurer le pic glycemique post-prandial
  • Recettes avec pic > 160 mg/dL → score penalise
  • Recettes avec pic < 120 mg/dL → score bonifie
```

**Timing** : phase 2, quand il y a assez de donnees d'usage. Le solveur LP est concu des la V1 pour accepter un score externe.

## 5. Socle deterministe

Ce qui ne passe **jamais** par un LLM :

### 4.1 Regles medicales (JSON as code)

```json

  "description": "TIR < 50% sur 7 jours → alerte praticien",
  "conditions": {
    "all": [
      { "fact": "tir_7d", "operator": "<", "value": 50 },
      { "fact": "cgm_readings_7d", "operator": ">=", "value": 500 }
    ]
  },
  "actions": [
    { "type": "alert_practitioner", "severity": "urgent", "reason": "tir_critically_low" },
    { "type": "add_message", "severity": "ALERT", "key": "msg.alerte.tir_bas" }
  ]
```

### 4.2 Exclusions

| Condition | Action | Deterministe |
|-----------|--------|---|
| IMC >= 40 | NON_COACHABLE, video exclusion | Oui, toujours |
| Diabetique type 1 | Exclusion, message adapte | Oui, toujours |
| Mineur (< 18 ans) | Exclusion | Oui, toujours |
| Enceinte | Exclusion | Oui, toujours |
| Maladie chronique, trouble alimentaire | Exclusion | Oui, toujours |
| Glycemie < 54 mg/dL (CGM) | Alerte urgente immediate | Oui, toujours |
| TIR < 50% sur 7j | Alerte praticien | Oui, toujours |
| GMI > 7.0% sur 14j | Recommandation consultation | Oui, toujours |

### 4.3 Solveur LP

L'optimisation de menus sous contraintes reste **mathematique** :
- Budget calorique (±10%)
- Contraintes alimentaires (casher, halal, gluten, vegetarien)
- Variete (max 1x/semaine par recette)
- Frequence par categorie d'aliment
- Saisonnalite
- Incompatibilites (casher : viande/laitier)

OR-Tools. Pas de LLM. Le LLM intervient apres pour **expliquer** le menu, pas pour le generer.

### 4.4 Calculs CGM

Les metriques TIR, TAR, TBR, GMI, CV sont des **calculs statistiques deterministes** (pandas/numpy). Le LLM intervient pour **interpreter** les resultats, pas pour les calculer.

```python
    checks = [

        check_no_medication_names(response),
```

## 8. Modele de programme adaptatif

### 7.1 Simplification des phases

Le legacy : 8 phases, 5 formules, 20 combinaisons, durees fixes, behaviors patchwork.

Glyc Pro : **4 phases, durees adaptatives**.

| Phase | Role | Duree | Critere de passage |
|-------|------|-------|--------------------|
| **DECOUVERTE** | Onboarding, baseline, premiers menus | 7-14 jours | Onboarding complete + 7j de donnees |
| **ACTION** | Perte de poids active, restriction calorique | 4-12 semaines | Objectif poids intermediaire atteint OU plateau > 3 semaines |
| **PROGRESSION** | Retour a l'alimentation normale | 4-12 semaines | Poids stable depuis 3 semaines + compliance > 70% |
| **STABILISATION** | Maintien, autonomie | 6-12 mois | Fin de programme |

### 7.2 Adaptation dynamique

Au lieu de durees fixes, le programme s'ajuste chaque semaine :

```python
def evaluate_phase_transition(patient: Patient) -> PhaseDecision:
    """Evalue chaque semaine si le patient doit changer de phase."""

    phase = patient.current_phase

    if phase == "DECOUVERTE":
        if patient.onboarding_complete and patient.days_of_data >= 7:
            return PhaseDecision(transition_to="ACTION")

    elif phase == "ACTION":
        weight_trend = patient.weight_trend_4w  # regression lineaire 4 semaines

        if patient.weight_kg <= patient.intermediate_target:
            return PhaseDecision(transition_to="PROGRESSION")

        if weight_trend.slope > -0.1 and patient.phase_weeks >= 3:
            # Plateau depuis 3+ semaines → passer en PROGRESSION
            return PhaseDecision(
                transition_to="PROGRESSION",
                reason="plateau_detected"
            )

        if patient.phase_weeks >= 12:
            # Max 12 semaines en ACTION
            return PhaseDecision(transition_to="PROGRESSION")

    elif phase == "PROGRESSION":
        weight_stable = abs(patient.weight_trend_3w.slope) < 0.05
        compliance_ok = patient.menu_compliance_4w >= 0.70

        if weight_stable and compliance_ok and patient.phase_weeks >= 4:
            return PhaseDecision(transition_to="STABILISATION")

    return PhaseDecision(stay=True)
```

### 7.3 Pas de formule rigide

Le legacy assignait F1-F5 selon Groupe × IMC. Glyc Pro simplifie :

| Parametre | Legacy | Glyc Pro |
|-----------|--------|----------|
| Ratio kcal phase ACTION | 0.8 fixe | 0.75-0.85 selon IMC et progression |
| Duree phase ACTION | 21-42j fixe selon formule | 4-12 semaines selon poids reel |
| Seances sport/semaine | Fixe selon GS1-GS4 | Progressif, ajuste selon compliance |
| Passage a STABILISATION | Behavior code en dur | Evaluation automatique chaque semaine |

Le scoring PT et la classification en groupes (G1/G2/G3) restent comme **indicateurs de risque initial**, mais ne determinent plus un parcours rigide.
